I§ Masterplan

一图看懂 — Agent 在中间，平台在两边。

前端 Chat 把意图发给 Agent；Agent 通过 MCP 调用后端工具；后端按平台分流 —— 有 API 的走云端 Provider，没 API 的走桌面端 Cookie 通道。

III§ Backend deep-dive

后端 — 两个进程，一条 MCP 总线。

把 AI Agent 和业务接口拆成两个独立进程：让 aitoearn-ai 可以重启不影响业务，让 aitoearn-server 单独水平扩展。

V§ The split

桌面端为什么存在 — 一个工程取舍。

小红书、视频号没有公开发布 API；抖音、快手、B站的 API 也覆盖不全。这些平台必须真人浏览器登陆。所以同一套 Agent 能力，被刻意切成两条物理通道。

CHANNEL · 桌面端通道
aitoearn-electron / plat/
必须本机 Cookie / WebView 才能上传

用户登录后 Cookie 留在本机，桌面端进程直接模拟浏览器请求平台。这些平台不在后端 providers/ 里。

抖音 快手 哔哩哔哩 小红书 视频号
vs.
CHANNEL · 云端 API 通道
aitoearn-server / providers/
有 OAuth + 开放 API，服务端直发

海外平台普遍提供 Open API；微信公众号虽在国内但也走 API。这些平台不在 electron/plat/ 里。

YouTube TikTok Instagram Facebook Threads X LinkedIn Pinterest Google Business 微信公众号

VIII§ Reading path

建议的二次阅读顺序

从契约 → Agent → 前端 → 桌面端。先建立"调用平台的统一抽象"心智，再看 Agent 怎么用它，最后理解物理拓扑。

iSTEP 01
读 Provider 统一契约
aitoearn-backend/apps/aitoearn-server/src/core/channel/publishing/providers/base.service.ts

理解 immediatePublish / finalizePublish 等生命周期方法，是所有平台 Provider 的共同形状。

iiSTEP 02
读 Agent 运行时
aitoearn-backend/apps/aitoearn-ai/src/core/agent/services/agent-runtime.service.ts

看 claudeQuery (197–370) 如何把 Claude 的 tool-use 协议翻译成对 MCP 工具的调用。

iiiSTEP 03
读前端流处理
aitoearn-web/src/store/agent/agent.methods.ts
+ task-instance/sse.handler.ts

看 handleStreamEvent 如何消费 Agent 的 SSE，并把工具调用渲染成卡片。

ivSTEP 04
看桌面端 Cookie 通道
aitoearn-electron/electron/plat/*

看小红书、视频号是怎么在 没有公开 API 的情况下被发布上去的。架构决策落地处。
